# 基于多智能体协同的媒体数据分析与知识库构建研究 —— 技术大纲

## 一、项目概述

本项目设计并实现一套以**多智能体协同编排**为核心的媒体数据分析与知识库构建系统。系统中每个 Agent 通过 **tool_call（函数调用）** 机制按需调用外部工具——包括爬虫工具（数据采集）与小模型工具（信息抽取）——各 Agent 围绕共享状态协同工作，对多源媒体数据进行采集、清洗、结构化抽取与关联融合，最终产出**结构化数据分析报告**与**可查询的领域知识库**。核心研究重心在于多 Agent 的任务编排、tool_call 工具调度、状态协同、冲突消解与结果融合。

## 三、多智能体协同编排层（核心模块 · 详细设计）

> 本章节为整个项目的核心，从 Agent 角色定义、编排框架、通信机制、状态管理、冲突消解到协同决策全链路展开。

### 3.1 Agent 角色体系

系统定义 **6 类专业化 Agent**，每类 Agent 采用 **LLM + tool_call** 模式运行——Agent 由 LLM 驱动决策，根据当前状态与任务目标，自主选择调用哪些工具（function calling），工具执行结果返回后 Agent 继续推理，形成"思考→调用工具→观察结果→再思考"的自主循环。每类 Agent 拥有独立的角色描述（Role）、目标（Goal）、注册工具集（Tools）与输出规范（Output Schema）：

#### 3.1.1 调度协调 Agent（Orchestrator）

| 维度 | 说明 |
|------|------|
| **核心职责** | 接收外部任务指令，解析意图，启动协同流程，汇总各 Agent 结果，触发报告生成 |
| **子能力** | 意图解析、任务可行性校验、全局流程编排、异常兜底 |
| **输入** | 用户/系统的分析请求（自然语言或结构化指令） |
| **输出** | 协同任务执行计划（DAG）+ 最终分析报告 |
| **不做什么** | 不直接操作数据、不直接调用 LLM 做内容分析（交由专职 Agent） |

#### 3.1.2 任务规划 Agent（Planner）

| 维度 | 说明 |
|------|------|
| **核心职责** | 将 Orchestrator 传来的高层意图分解为可执行的原子任务 DAG，定义任务间依赖与并行策略 |
| **分解逻辑** | 按数据链路（采集→清洗→分析→建模→质检）纵向切分 + 按数据源/实体维度横向并行 |
| **输入** | 结构化的任务意图描述 |
| **输出** | 任务 DAG（节点 = 原子任务，边 = 依赖关系），附带每节点的目标 Agent 类型、优先级、预估耗时 |
| **关键算法** | 基于拓扑排序的任务分层 + 贪心并行度最大化 |

**任务 DAG 示例：**
```

                    │  AI 行业热点事件"    │

                             │
              ┌──────────────▼──────────────┐
              │    采集任务（3 数据源并行）    │
              │  RSS源 │ Twitter │ 新闻网站   │
```

#### 3.1.3 采集管理 Agent（Collector）

| 维度 | 说明 |
|------|------|
| **核心职责** | 根据 Planner 下发的采集子任务，通过 tool_call 调度爬虫工具执行数据抓取 |
| **注册工具集** | RSS 抓取工具、动态网页爬虫工具（底层 Scrapy + Playwright）、社交媒体 API 检索工具、正文清洗工具 |
| **tool_call 决策逻辑** | LLM 根据数据源 URL 特征自动选择工具——RSS 链接调用 RSS 工具、动态页面调用爬虫工具、API 端点调用检索工具；抓取完成后自动调用清洗工具做正文抽取 |
| **自主决策** | 根据工具返回的状态码/异常，自主决定重试、切换备用源或降低抓取频率 |
| **输出** | 标准化原始数据条目列表（JSON Schema 约束），写入 SharedState |

#### 3.1.4 分析抽取 Agent（Analyzer）

| 维度 | 说明 |
|------|------|
| **核心职责** | 对清洗后的文本，通过 tool_call 调用小模型工具与 LLM 工具执行 NLP 信息抽取 |
| **注册工具集** | 实体识别工具（调用 HanLP / BERT 小模型本地推理）、关系抽取工具（调用 LLM 少样本抽取）、事件抽取工具（触发词检测 + 论元标注）、摘要生成工具（LLM 生成式摘要） |
| **tool_call 决策逻辑** | LLM 分析文本特征后分步调用——先调用小模型实体识别（低延迟、零 API 成本），再对有实体的句子调用 LLM 关系抽取与事件抽取，最后调用摘要生成；短文本跳过事件抽取 |
| **自主决策** | 根据文本长度/复杂度分级策略——短文本全量 LLM 抽取，长文本先经小模型粗筛，仅对高价值段落调用 LLM；小模型置信度不足时自动升级为 LLM 调用 |
| **输出** | 结构化抽取结果（实体列表、三元组列表、事件卡片、摘要），附带每字段置信度 |

#### 3.1.5 知识建模 Agent（Knowledge Modeler）

| 维度 | 说明 |
|------|------|
| **核心职责** | 将分析 Agent 产出的结构化信息，通过 tool_call 调用图谱工具融合到知识库 |
| **注册工具集** | 图谱查询与更新工具（Cypher 执行）、实体链接工具（向量相似匹配 + 消歧）、关系融合工具（跨源合并去重）、子图上下文检索工具 |
| **tool_call 决策逻辑** | 对每个新实体先调用实体链接工具判定是否与已有实体指代同一对象，再调用图谱更新工具执行 MERGE/CREATE；对每条新关系调用关系融合工具检查冲突 |
| **自主决策** | 实体消歧三级漏斗——向量相似初筛 → 图谱结构匹配 → LLM 终判；关系冲突时回退至 Reviewer 仲裁 |
| **输出** | 图谱更新操作日志、新增/变更的实体与关系清单 |

#### 3.1.6 质量审核 Agent（Reviewer）

| 维度 | 说明 |
|------|------|
| **核心职责** | 对全链路产出进行质量校验，通过 tool_call 调用校验工具与 LLM Judge 进行裁决 |
| **注册工具集** | Schema 校验工具（字段完整性检查）、冲突检测工具（实体/关系矛盾识别）、LLM 仲裁工具（冲突最终裁定）、置信度过滤工具（低质量结果标记） |
| **tool_call 决策逻辑** | 先调用 Schema 校验逐字段检查完整性，再调用冲突检测工具发现矛盾，发现冲突后调用 LLM 仲裁做最终裁定，最后调用置信度过滤标记低质量条目 |
| **自主决策** | 低置信度结果标记为待复核或触发重新分析；多次修正仍不达标则丢弃并记录；不可自动裁决的冲突升级至人工 |
| **输出** | 质量评估报告 + 修正建议 + 异常条目清单 |

### 3.2 编排框架设计

采用 **LangGraph** 作为核心编排框架，理由如下：
- 原生支持**有状态图（StateGraph）**，适合多 Agent 流水线的状态传递
- 支持**条件路由**——根据中间结果动态决定下一步执行路径
- 内置**人在回路（Human-in-the-Loop）**中断点，便于质检 Agent 拦截低质量结果
- 图结构天然映射任务 DAG，可观测性强

**编排层的核心数据结构 —— 共享状态（SharedState）：**

```python
# 概念定义（非具体代码）
SharedState = {
    # 任务元信息
    "task_id": str,              # 全局唯一任务 ID
    "intent": str,               # 初始用户意图
    "status": Enum,              # PENDING / RUNNING / SUCCESS / FAILED

    # 任务规划
    "plan": DAG,                 # Planner 产出的任务 DAG
    "current_stage": str,        # 当前执行阶段标识

    # 数据流
    "raw_documents": List[Doc],  # 采集产出（各 Agent 追加写入）
    "cleaned_documents": List[Doc],
    "extracted_entities": List[Entity],
    "extracted_relations": List[Relation],
    "extracted_events": List[Event],

    # 质量元数据
    "confidence_scores": Dict,   # 各步骤置信度
    "review_flags": List[Flag],  # 质检标记

    # 最终产出
    "report": str,               # 最终分析报告
    "knowledge_updates": Dict,   # 知识库变更摘要

    # 持久化与恢复
    "checkpoint_id": str,        # 状态检查点 ID，用于断点恢复
}
```

**编排图（LangGraph StateGraph）节点定义：**

```
START
  │
  ▼
[plan_node]          —— Planner Agent：意图 → 任务 DAG
  │
  ▼
[dispatch_node]      —— Orchestrator：遍历 DAG 就绪节点，并行分发
  │
  ├──→ [collect_node]   —— Collector Agent × N（并行采集）
  ├──→ [clean_node]     —— 清洗节点
  ├──→ [analyze_node]   —— Analyzer Agent × N（并行分析）
  ├──→ [model_node]     —— Knowledge Modeler Agent
  │
  ▼
[review_node]        —— Reviewer Agent：质量校验
  │
  ├── (通过) → [merge_node]    —— 结果融合 + 知识入库
  │                │
  │                ▼
  │           [report_node]    —— 报告生成
  │                │
  │                ▼
  │              END
  │
  └── (不通过) → [correct_node] —— 修正回路（回退到 analyze/model）
                     │
                     └──→ review_node（重新审核）
```

**条件路由逻辑：**
- `review_node` 输出置信度 ≥ 阈值 → 走 `merge_node`
- `review_node` 发现冲突/低质量 → 走 `correct_node`，携带修正指令回退到对应阶段
- 回退次数超过上限 → 标记为需人工介入，仍产出报告但附带不确定性标注

### 3.3 Agent 通信与状态同步

Agent 间通过 **LangGraph StateGraph 共享状态** 实现数据流转——每个节点执行完毕后自动更新 State，下游节点从 State 中读取上游产出。并行 Agent 间通过 State 的原子写入保证数据一致性。

| 机制 | 实现方式 | 适用场景 |
|------|----------|----------|
| **共享状态传递** | LangGraph StateGraph 原生 State 在节点间自动流转 | 上下游串行任务的数据传递 |
| **直接调用** | Agent 间的 Tool 化封装，A Agent 可将 B Agent 作为工具调用 | 紧密耦合的子任务委托 |
| **状态持久化** | State 定期写入 MySQL，支持断点恢复与执行审计 | 长任务的状态检查点 |

---

### 3.4 协同决策机制

当多个 Agent 对同一数据产生不同判断时，采用以下分层决策策略：

| 冲突类型 | 决策机制 | 说明 |
|----------|----------|------|
| **实体消歧冲突** | 向量相似度（初筛）→ 图谱上下文比对 → LLM 仲裁（终判） | 三级漏斗式消歧 |
| **关系冲突**（A 说相关，B 说不相关） | 置信度加权投票 + 证据链审查 | Reviewer Agent 汇总各方证据后裁定 |
| **事实冲突**（同一事件的不同描述） | 来源权威度加权 + 时效性优先 + 多数源一致性 | 附置信度标注，保留多源说法 |
| **任务优先级冲突** | Orchestrator 集中决策，基于紧迫度 + 依赖深度排序 | 避免各 Agent 自行抢占资源 |

**LLM 仲裁 Prompt 设计思路：**
- 输入：冲突的两个候选 + 各自的证据文本 + 图谱上下文
- 输出：选中的候选 + 置信度 + 判定理由
- 约束：必须二选一，不可产生新实体

## 四、Agent 工具层 —— 可被 tool_call 调用的工具集合

> 本章节描述各 Agent 通过 tool_call 调用的底层工具实现。每个工具被封装为标准的 Function Calling Schema（函数名 + 参数定义 + 返回值类型），Agent 的 LLM 根据任务上下文自主决策调用哪些工具。

### 4.1 爬虫工具（Collector Agent 调用）

Collector Agent 通过 tool_call 调用爬虫工具执行多源数据抓取。每个工具封装为独立的 function，底层基于 **Scrapy + Playwright** 实现动态页面渲染与反爬对抗，内置 **SimHash + URL 精确匹配**双重去重。

| 工具类型 | 说明 |
|----------|------|
| RSS 抓取工具 | 解析 RSS/Atom 订阅源，批量提取文章标题、正文、发布时间 |
| 动态网页爬虫 | Playwright 渲染 JS 页面，支持自定义 CSS 选择器精准抽取 |
| API 检索工具 | 调用社交媒体 API（微博/Reddit 等），按关键词与时间范围检索 |
| 正文清洗工具 | trafilatura / Readability 提取 HTML 正文，分离标题-正文-时间-来源四要素 |

**关键实现点：**
- 每个工具函数标注 `@tool` 装饰器，自动生成 OpenAI-compatible Function Calling Schema
- Collector Agent 的 LLM 根据 URL 特征自动选择工具——RSS 链接调用 RSS 工具、动态页面调用爬虫工具
- 抓取完成后 LLM 自动判断是否需要调用清洗工具做正文抽取

### 4.2 NLP 工具（Analyzer Agent 调用）

Analyzer Agent 通过 tool_call 分层调用小模型与 LLM 工具完成信息抽取——先以**小模型本地推理**做粗筛，再按需调用 LLM 工具精抽，降低 API 成本：

| 工具类型 | 底层方案 | 说明 |
|----------|----------|------|
| 实体识别工具 | HanLP / BERT 小模型本地推理 | 6 类命名实体识别（PER/ORG/LOC/TIME/EVENT/TOPIC），零 API 成本 |
| 关系抽取工具 | LLM Prompt 少样本抽取 | 对含实体的句子抽取主-谓-宾三元组 |
| 事件抽取工具 | 规则触发词 + LLM 论元标注 | 识别事件及其参与方、时间、地点 |
| 摘要生成工具 | LLM 生成式摘要 | 全文摘要，保留关键信息 |

**分层调用策略（成本优化核心）：**
```
文本输入
  │
  ├──→ 第一步：实体识别（小模型本地推理，零 API 成本）
  │        仅对有实体的句子继续后续处理
  │
  ├──→ 第二步：关系抽取（LLM，仅处理含实体的句子）
  │
  ├──→ 第三步：事件抽取（含时间/地点实体的文本才触发）
  │
  └──→ 第四步：摘要生成（LLM，全文本摘要）
```

**关键实现点：**
- 小模型工具在 Schema 中标注低成本标签，LLM 可见成本提示，优先选择
- Analyzer Agent 根据文本长度自主选择调用链——短文本全量调用，长文本先小模型粗筛再 LLM 精抽
- 小模型置信度不足时自动升级为 LLM 调用并标记，确保准确率

### 4.3 知识库工具（Knowledge Modeler + Reviewer 调用）

| 工具类型 | 底层方案 | 说明 |
|----------|----------|------|
| 图谱查询与更新工具 | Neo4j Python Driver | 执行 Cypher 查询，MERGE/CREATE 实体与关系 |
| 实体链接工具 | 向量相似 + 图谱结构匹配 | 判断新实体是否与已有实体指代同一对象 |
| 关系融合工具 | 规则 + LLM 判断 | 跨源关系合并去重，冲突检测 |
| Schema 校验工具 | JSON Schema 校验器 | 逐字段检查数据完整性与类型合规 |
| LLM 仲裁工具 | LLM Judge | 实体消歧与关系冲突的最终裁定 |

## 五、知识库构建模块

### 5.1 MySQL —— 结构化业务数据

| 数据表 | 存储内容 |
|--------|----------|
| `raw_articles` | 采集的原始媒体数据（标题、正文、URL、发布时间、来源） |
| `cleaned_articles` | 清洗后的标准化数据 |
| `entities` | 抽取的实体索引（名称、类型、首次发现时间） |
| `relations` | 关系记录（头实体、尾实体、关系类型、置信度） |
| `analysis_reports` | 生成的分析报告归档（Markdown/JSON） |
| `agent_execution_log` | Agent 执行日志（task_id、agent_type、action、结果、耗时） |
| `quality_reviews` | 质检记录（reviewer 判定、置信度、修正动作） |

### 5.2 Neo4j —— 知识图谱

**图谱 Schema 设计：**

| 节点标签 | 属性 | 说明 |
|----------|------|------|
| `Entity` | name, type, aliases, first_seen, confidence | 实体节点（PER/ORG/LOC/EVENT/TOPIC） |
| `Article` | title, url, publish_time, source | 媒体文章节点 |
| `Report` | title, created_at, summary | 分析报告节点 |

| 关系类型 | 方向 | 属性 | 说明 |
|----------|------|------|------|
| `MENTIONS` | Article → Entity | count, confidence | 文章提及某实体 |
| `RELATED_TO` | Entity → Entity | relation_type, confidence, evidence | 实体间语义关系 |
| `HAPPENED_AT` | Entity(Event) → Entity(LOC) | time | 事件发生地点 |
| `INVOLVES` | Entity(Event) → Entity(PER/ORG) | role | 事件参与方 |
| `BELONGS_TO` | Entity → Entity(TOPIC) | — | 实体归属主题 |
| `DERIVED_FROM` | Report → Article | — | 报告引用文章 |

**典型 Cypher 查询能力：**
- 实体关联网络：`MATCH (e:Entity {name:"X"})-[r:RELATED_TO]-(other) RETURN e, r, other`
- 事件时间线：`MATCH (e:Entity {type:"EVENT"}) WHERE e.time > ... RETURN e ORDER BY e.time`
- 热点发现：`MATCH (a:Article)-[:MENTIONS]->(e:Entity) WHERE a.publish_time > ... RETURN e.name, COUNT(*) AS freq ORDER BY freq DESC LIMIT 20`

### 5.3 图谱与结构化数据的协同

```
MySQL（元数据 + 事实表）
    ↕  双向关联（entity_id / article_id 作为关联键）
Neo4j（实体-关系网络）

查询路径：
  用户查询意图
      │
      ├──→ MySQL：按时间/来源/主题过滤文章、读取报告
      │
      └──→ Neo4j：实体关联扩展、路径发现、子图探索
              │
              └──→ 汇总结果 → LLM 生成分析报告
```

## 六、分析报告生成

系统最终产出为**结构化数据分析报告**，分为以下类型：

| 报告类型 | 内容 | 触发方式 |
|----------|------|----------|
| **热点事件报告** | 指定时间段内的热点事件 Top-N，含事件摘要、关联实体、传播路径 | 定时/按需 |
| **实体深度分析** | 某实体的关联网络、相关事件时间线、情感倾向演变 | 按需 |
| **主题趋势报告** | 某主题下的事件趋势、关键参与方变化、舆论走向 | 定时/按需 |
| **数据质量报告** | 采集覆盖度、抽取准确率、知识冲突统计 | 定时 |

**报告生成流程：**
```
Orchestrator 接收报告请求
    → Planner 分解报告所需的数据子任务
    → 各 Agent 并行执行（采集 + 分析 + 建模）

    → Orchestrator 汇总各维度结果
    → LLM 基于结构化数据撰写报告（带来源引用）
    → 输出 Markdown + JSON（Markdown 供人阅读，JSON 供程序消费）
```

## 十二、关键技术难点与应对

| 难点 | 应对策略 |
|------|----------|
| **Agent 协同的状态一致性** | 以 LangGraph StateGraph 为单一状态源，State 在每节点执行后原子更新，避免多 Agent 直接并发写同一字段 |
| **任务 DAG 的动态调整** | Planner 产出的 DAG 非一次性固化——若某分支失败，Planner 可被重新触发以生成降级计划 |
| **实体对齐的准确性** | 三级漏斗消歧：向量相似初筛（粗）→ 图谱结构匹配（细）→ LLM 仲裁（终），每一级过滤候选项 |
| **LLM 调用成本与延迟** | 分层策略：小模型本地推理做粗筛（NER 预标注），LLM 仅处理高难度子集（关系抽取、报告撰写）；结果缓存避免重复调用 |
| **知识时效性衰减** | 实体/关系带时间衰减权重，超期数据标记为"历史快照"，新分析优先使用近期数据 |
| **tool_call 工具选择准确性** | Agent Prompt 中声明各工具的成本与适用场景，LLM 据此自主决策调用顺序；工具返回结构化错误码辅助 Agent 判断重试或切换 |
